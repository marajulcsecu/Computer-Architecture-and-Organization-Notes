Topic 1: CPU Performance Equation

CPU Performance Equation: CPU time = Instruction Count (IC) x Cycles Per Instruction (CPI) x Clock Cycle Time
CPU time = IC x CPI x Clock cycle time CPU time = (IC x CPI) / Clock rate CPU clock cycles = IC x CPI

Meaning of each term

Term Meaning Typical unit
IC Total instructions executed by the program instructions
CPI Average cycles needed per instruction cycles/instruction
Clock cycle time Time duration of one clock cycle seconds/cycle
Clock rate Number of cycles per second Hz (cycles/sec)

Unit proof: Why "seconds per program"

CPU time = (instructions) x (cycles/instruction) x (seconds/cycle) = seconds

Visual formula map

Numerical example

Suppose IC = 2,000,000 instructions, CPI = 2.5, and clock rate = 500 MHz.

CPU time = (IC x CPI) / Clock rate = (2,000,000 x 2.5) / (500 x 10^6) = 5,000,000 / 500,000,000 = 0.01 s = 10 ms
Exam Focus:
  • Always write both forms: with cycle time and with clock rate.
  • Use unit-cancellation line to justify seconds/program.
  • Do not confuse CPI with clock rate; one is per instruction, one is per second.

Topic 2: MIPS, Throughput, and Processor Performance Comparison

MIPS: Million Instructions Per Second. It indicates instruction execution rate, not absolute quality of processor for all workloads.

MIPS formulas

MIPS = Instruction count / (Execution time x 10^6) MIPS = Clock rate / (CPI x 10^6)

Throughput

Throughput = Number of completed jobs / Total time

Higher throughput means system handles more tasks per unit time.

Performance comparison rule

Performance is inversely proportional to execution time. If execution time decreases, performance increases.

When MIPS can mislead

  • Different ISAs use different instruction complexity.
  • Higher MIPS does not always mean lower program time.
  • Always combine MIPS with CPI, clock rate, and real execution time.

Exam Focus:
  • For comparisons: compute instructions/sec first, then cycles and instructions if needed.
  • Always mention why MIPS alone can be incomplete.

Topic 3: Amdahl Law and Overall Speedup Interpretation

Amdahl Law: Overall speedup depends on both improved part and unimproved part. Improving only one portion of a program has limited global impact.

Main formula (single enhanced fraction)

Speedup_overall = 1 / ((1 - f) + f/S) where f = fraction enhanced, S = speedup of enhanced fraction

Alternative time-based method (for multiple fractions)

T_new = Sum(T_i / S_i for improved parts) + Sum(T_j for unchanged parts) Overall speedup = T_old / T_new

Interpretation points

  • If f is small, even very large S gives small overall gain.
  • As S -> infinity, maximum overall speedup = 1/(1-f).
  • So bottleneck (unimproved portion) sets upper performance limit.

Topic 4: Numerical Speedup Problems with Partial Enhancements

Standard solving method

  1. Convert percentages into time portions using total execution time.
  2. Convert each enhancement to speedup factor S (e.g., 30% faster -> S = 1.3).
  3. Compute each improved part as old_time / S.
  4. Add unchanged parts.
  5. Overall speedup = old total / new total.

Worked example from teacher resource

Program time = 100 ms with A=10 ms, B=30 ms, C=60 ms. A and B are each improved by factor 2.

T_new = 10/2 + 30/2 + 60 = 5 + 15 + 60 = 80 ms Speedup = 100/80 = 1.25

Common interpretation trap

If question says "enhanced 25%", default exam interpretation is usually speedup factor 1.25, not time reduction directly by 25%. Mention your assumption briefly in solution for safety.

Topic 6: Reliability Terms - MTTF, FIT, MTBF

Reliability analysis estimates failure behavior and service continuity of hardware/system over time.
Term Full form Meaning
MTTF Mean Time To Failure Average operating time before failure (non-repairable view)
FIT Failures In Time Number of failures per 10^9 device-hours
MTBF Mean Time Between Failures Average interval between failures in repairable systems
MTTR Mean Time To Repair Average downtime for repair and restoration

Core relationships

Failure rate lambda = 1 / MTTF FIT = 10^9 / MTTF(hours) MTBF = MTTF + MTTR (repairable system model) Availability = MTTF / (MTTF + MTTR)

Reliability conversion example

Given MTTF = 200,000 hours:

FIT = 10^9 / 200,000 = 5000 FIT

Availability example

If MTTF = 10,000 hours and MTTR = 10 hours:

Availability = 10000 / (10000 + 10) = 0.999001 approx 99.90%

Topic 7: Bottleneck Analysis and Performance Trade-Offs

Bottleneck: The slowest or most constrained component limits total system performance.

Performance bottleneck examples

  • Fast CPU but slow memory access (memory wall).
  • High clock rate but large CPI due to stalls.
  • High compute speed but poor I/O throughput.

Trade-off table

Design decision Benefit Possible trade-off
Increase clock rate Can reduce cycle time Higher power/heat, may increase CPI due to deeper pipelines
Larger cache Lower miss rate Higher cost, larger access latency after some point
Add more cores Higher throughput Parallel software limit, memory contention
Very aggressive optimization of one part Local speedup Diminishing overall return (Amdahl limit)

Exam answer template for bottleneck questions

  1. Identify the slowest component using timing numbers.
  2. Apply Amdahl style limit to show global impact.
  3. Suggest balanced improvement (CPU + memory + I/O together).
  4. Mention trade-offs: cost, power, complexity, reliability.
Exam Focus:
  • Words like "balance" and "system-level optimization" are important for marks.
  • Always connect bottleneck discussion to quantitative speedup limit.

PYQ 2020 Sec A(a) / 2023 Q1(b): State Amdahl law for system performance enhancement.

Model answer:

Amdahl law states that overall system speedup due to enhancement of one component is limited by the fraction of total execution time that component originally consumes. Even infinite improvement of one part cannot remove time spent in unaffected parts.

Speedup_overall = 1 / ((1 - f) + f/S)

Here, f is the fraction improved and S is speedup factor of that fraction. This law is used to estimate realistic performance gain before optimization.

PYQ 2019 Q2(b) Part 1: 65 ms process, parts a/b/c = 60/30/10%, b enhanced 25%, c enhanced 20%.
Step 1: Original time split
a = 0.60 x 65 = 39 ms
b = 0.30 x 65 = 19.5 ms
c = 0.10 x 65 = 6.5 ms
Step 2: Apply enhancement factors
Assume "enhanced 25%" => speedup factor 1.25, and "enhanced 20%" => speedup factor 1.20.
b_new = 19.5 / 1.25 = 15.6 ms
c_new = 6.5 / 1.20 = 5.4167 ms
Step 3: New total time and speedup
T_new = 39 + 15.6 + 5.4167 = 60.0167 ms
Speedup = 65 / 60.0167 = 1.083

Final answer: Overall speedup approx 1.083, i.e., about 8.3% improvement.

PYQ 2019 Q2(b) Part 2: Processor 10x faster on computation, original time 40% compute and 60% I/O wait.
Speedup = 1 / ((1 - 0.4) + 0.4/10) = 1 / (0.6 + 0.04) = 1 / 0.64 = 1.5625

Final answer: Overall speedup = 1.5625 (approx 56.25% faster than original).

PYQ 2022 Q2(b): 100 ms process, a/b/c = 70/20/10%, b enhanced 30%, c enhanced 20%.
Original times: a=70 ms, b=20 ms, c=10 ms.
Enhancements interpreted as speedup factors: b=1.3, c=1.2.
b_new = 20/1.3 = 15.3846 ms
c_new = 10/1.2 = 8.3333 ms
T_new = 70 + 15.3846 + 8.3333 = 93.7179 ms
Speedup = 100/93.7179 = 1.067

Final answer: Overall speedup approx 1.067 (about 6.7% improvement).

PYQ 2023 Q2(b): 50 ms process with f1/f2/f3/f4 = 50/30/15/5%, f1 2x faster, f2 3x faster.
f1 = 25 ms, f2 = 15 ms, f3 = 7.5 ms, f4 = 2.5 ms.
New times: f1_new = 25/2 = 12.5 ms, f2_new = 15/3 = 5 ms, f3 and f4 unchanged.
T_new = 12.5 + 5 + 7.5 + 2.5 = 27.5 ms
Speedup = 50/27.5 = 1.818

Final answer: Overall speedup approx 1.818 (about 81.8% faster).

PYQ 2022 Q2(c): Explain CPU clock cycles = Sum(IC_i x CPI_i).

Model explanation:

A program has multiple instruction classes (for example arithmetic, branch, load/store). Each class has instruction count IC_i and per-instruction cycle requirement CPI_i. Total cycles needed by class i = IC_i x CPI_i. Summing all classes gives total clock cycles of entire program:

Total cycles = Sum(IC_i x CPI_i)

Therefore the equation is fundamental for CPI-based performance analysis and is directly used to compute execution time: CPU time = Total cycles / Clock rate.

PYQ 2023 Q2(c): Show CPU time = Seconds/program from CPU time equation.

Model answer:

CPU time = Instruction count x Cycles per instruction x Seconds per cycle
Units: (instructions) x (cycles/instruction) x (seconds/cycle) = seconds

Since instruction and cycle units cancel out, the final unit is seconds required to run one program, i.e., seconds/program.

PYQ 2020 Sec B(d): Compare P1/P2/P3 and compute required clock for 30% time reduction with 20% CPI increase.
Given:
P1: 3.0 GHz, CPI 1.5 -> IPS = 3.0e9/1.5 = 2.0e9 instr/s
P2: 2.5 GHz, CPI 1.0 -> IPS = 2.5e9 instr/s
P3: 4.0 GHz, CPI 2.2 -> IPS = 1.818e9 instr/s

(i) Highest performance: P2.

(ii) If each runs for 10 sec:
P1 cycles = 3.0e9 x 10 = 30e9, instructions = 30e9/1.5 = 20e9
P2 cycles = 2.5e9 x 10 = 25e9, instructions = 25e9/1.0 = 25e9
P3 cycles = 4.0e9 x 10 = 40e9, instructions = 40e9/2.2 = 18.1818e9
(iii) New clock with 30% less time and 20% more CPI:
Time_new = 0.7 Time_old, CPI_new = 1.2 CPI_old
Clock_new/Clock_old = (1.2/0.7) = 1.7143

So required clock is 1.7143 times old clock. Example for P1: 3.0 x 1.7143 = 5.14 GHz.

PYQ 2023 Q3(a): Find MIPS rate (2 million instructions, 400 MHz, given CPI mix).
Weighted CPI: CPI_avg = 0.60x1 + 0.18x2 + 0.12x4 + 0.10x8 = 0.6 + 0.36 + 0.48 + 0.8 = 2.24
Instruction rate = Clock rate / CPI_avg = 400e6 / 2.24 = 178.571e6 instructions/s MIPS = 178.571

Final answer: MIPS rate approx 178.57.

PYQ 2022 Q1(a): "Processing speed has increased more rapidly than memory access time" - justify.

Model answer:

CPU technology scaling increased core frequency and parallel processing capability much faster than DRAM latency reduction. So relative latency gap increased. Example: CPU cycle can reduce from 10 ns to 1 ns while memory access may reduce from 100 ns to 60 ns. Ratio changes from 10:1 to 60:1. Hence CPU waits proportionally more for memory despite absolute improvement.

This is the memory bottleneck (memory wall), motivating cache hierarchy and locality-aware optimization.
